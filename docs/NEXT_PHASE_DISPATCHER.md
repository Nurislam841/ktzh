# Next Phase: Dispatcher-Grade Node Automation

## Что уже добавлено

- На экране узла появилась очередь решений диспетчера.
- Система показывает ближайшие отправления с проблемами по:
  - бригаде (`T-2 часа`);
  - приемке локомотива (`T-1 час`);
  - пути;
  - накопленной задержке.
- Backend endpoint: `GET /node/decision-queue?stationId=...&hours=6|12|24`
- Реализован базовый контур `CrewCall`:
  - `GET /crew-calls`
  - `POST /crew-calls/generate`
  - `PATCH /crew-calls/:id/status`
- В `TrainRun` добавлены сценарии:
  - `FORMATION`
  - `TRANSIT`
  а также флаги `requiresCrewChange` и `requiresLocoChange`

## Что делать следующим шагом

### 1. Довести доменную модель до операционной полноты

Базовые поля уже добавлены в `TrainRun`. Следующий шаг:

- `operationScenario`: `FORMATION | TRANSIT`
- `requiresCrewChange`: `boolean`
- `requiresLocomotiveChange`: `boolean`
- `targetTrackId`: `nullable`

Это позволит отделить:

- формирование состава с нуля;
- прием транзитного поезда с быстрой сменой ресурсов.

### 2. Развить контур вызова бригады

Базовая сущность `CrewCall` уже добавлена. Следующий шаг:

- `trainRunId`
- `crewId`
- `mustReportAt`
- `acceptedLocomotiveAt`
- `status`: `PLANNED | NOTIFIED | CONFIRMED | MISSED`

Для промышленного контура нужен полный жизненный цикл:

- кто уведомил бригаду;
- когда подтверждена явка;
- когда принят локомотив;
- почему произошел срыв.

### 3. Улучшить решатель

Текущий `greedy-solver` нужно перевести с модели:

- "один локомотив/бригада используются один раз"

на модель:

- "ресурс имеет окно доступности и может пойти на следующий рейс после завершения предыдущего".

Минимальный безопасный шаг:

- заменить `usedLocomotiveIds` и `usedCrewIds` на карты доступности `resourceId -> nextAvailableAt`;
- учитывать `plannedArrival` предыдущего назначения;
- использовать правила `T-2/T-1` уже в самом выборе бригады, а не только в аналитике.

### 4. Добавить участок/перегон

Сейчас система сильна на уровне станции. Для сценариев "остановка на перегоне" нужны:

- сущность `Section`;
- занятость участка по времени;
- базовый headway между станциями.

### 5. Сделать экран решений

После `decision-queue` логичное следующее UI-расширение:

- `Decision Detail`
- причина проблемы;
- альтернативы A/B/C;
- ожидаемая задержка по каждому варианту;
- влияние на соседние поезда.

## Следующие API

- `GET /node/decision-queue`
- `GET /node/decision-queue/:allocationId`
- `POST /node/decision-queue/:allocationId/ack`
- `POST /crew-calls/generate`
- `PATCH /crew-calls/:id/status`
- `POST /optimizer/station/:id/rebuild`

## Критерий готовности следующей фазы

- Два сценария поддерживаются явно: `FORMATION` и `TRANSIT`.
- Бригада живет как отдельный процесс вызова и подтверждения.
- Решатель умеет переиспользовать локомотивы и бригады по времени.
- Система объясняет диспетчеру не только проблему, но и лучший вариант действия.
